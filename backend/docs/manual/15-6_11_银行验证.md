# 6.11、银行验证

一、模块概述

1.1 来源与上下游

上游来源：注册表模块，由注册表【审核通过】后按业务规则自动推送单据至本「银行验证管理」；

下游分 3 种流向：

注册类型 =【店铺 / 支付】：银行审核通过 → 自动推送至【店铺校验管理】；

注册类型 =【银行】：银行审核通过 → 直接流入【账号部→银行卡管理】，跳过店铺验证、不进入账号管理；

银行审核驳回：单据退回注册表【审核中】状态，由注册员工重新修改资料。

触发进银行验证前置条件：订单池建档勾选「需要银行资料」+ 注册表审核通过。

1.2 归属与使用角色

左侧菜单栏：账号部→银行验证管理；操作人员：银行审核专员、平台注册部管理员、超级 Admin。

二、单据进件规则（核心分流逻辑）

2.1 推送至本页面的判定条件

源头：订单池录入资料时勾选【需要银行】；

注册表流程：管理员底表资料审核通过；

按【注册类型】区分 3 类单据：

| 注册类型 | 流转规则 |

| ---- | ---- |

| 店铺 / 支付 | 注册表→银行验证→店铺校验管理→店铺审核通过后入【账号管理（库存）】|

| 银行 | 注册表→银行验证→银行卡管理，跳过店铺验证、不进账号管理 |

| 免银行（订单池未勾选需要银行）| 注册表审核通过直接跳店铺验证，不经过银行验证模块|

2.2 初始状态

注册表推送后，银行验证单据默认待核验状态，带出注册表基础字段：平台、属地、法人、企业名、属性、项目编号、订单编号等。

三、页面功能说明（参考原型图）

3.1 顶部筛选查询区

可组合条件检索单据模糊输入框：项目编号、公司名称、备注

下拉筛选：平台、属地、店铺种类、单据状态

按钮：搜索、重置（清空全部查询条件）

3.2 列表展示字段

申请人、申请时间、平台、属地、属性、法人、公司、店铺种类；

店铺种类标签区分：【店铺 / 支付】，用来快速区分后续流转去向。

3.3 弹窗【银行校验】（点击单据打开右侧弹窗）

1. 基础回显（只读，来自上游注册表）

平台、法人、法人姓名、法人生日（不可编辑）

2. 银行核心必填项

银行卡类型（单选：个人卡 / 企业卡 / 第三方卡，必选）

银行名称、银行卡号、户名（必填项 *）

银行卡状态（开关：可用 / 不可用）

3. 辅助信息字段

邮箱链接、邮箱账号、邮箱密码；信用卡号、联系电话；支行、支行号；备注文本框

4. 申请人附件区：查看注册员工上传的银行相关附件底表

5. 弹窗操作按钮：审核通过 / 审核驳回（填写驳回备注）

四、银行审核操作规则

4.1 审核通过

注册类型 = 店铺 / 支付：单据从银行验证办结，自动推送至【店铺校验管理】待办列表；

注册类型 = 银行：单据办结，数据直接同步至【账号部→银行卡管理】模块生成银行卡档案，终止流程，不流转店铺、不进账号管理（库存）。

4.2 审核驳回

填写驳回原因，单据原路退回注册表【审核中】状态，注册表待办提醒注册经办人补充银行资料后重新提交审核。

五、状态与统计规则

待核验：注册表已推送、银行未打开校验弹窗；

审核通过：已办结并向下游对应模块推送；

审核驳回：退回注册表；

页面左侧菜单角标数字：代表当前待审核单据总量（原型数字 1）。
